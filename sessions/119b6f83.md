## Technical Context

**Architecture decisions**:
- **Extended thinking**: Only `adaptive` vs `off` (no `enabled` mode). `adaptive` preserves cache, supports interleaved thinking. Temperature forced to 1.0 when thinking on.
- **Thinking block signatures**: Cryptographically signed by API. Text must match signature exactly. Delta accumulation → use SDK canonical text at `ContentBlockStop`.
- **Thinking block pruning**: Old thinking blocks stripped in prune phase (before protected boundary). API auto-ignores old ones in context calculations anyway.
- **Cache invalidation**: Thinking mode switches break message cache (but system/tool cache preserved). Per-agent defaults avoid mid-session switches.
- **Compaction overflow**: If pruning removes <10%, force summarization (prune ratio > 0.9 is ineffective).
- **Cron session reuse**: Persist session IDs to `~/.nous/cron_state.json`, restore on daemon restart.
- **TodoList restoration**: Replay all `add`/`batch_add`/`done`/`delete` calls from session messages on `buildLS`.

## Strategy & Approach

**Overall approach**: Implement extended thinking with cache stability, fix critical bugs (thinking signature mismatches, compaction overflow, discuss transcript ordering), improve UX (session hiding, error details, auto-resume).

**Why this approach**:
- **Thinking mode stability**: Per-agent defaults + slash command for manual toggle. Avoid mid-session switches (cache penalty).
- **Thinking block fidelity**: Use SDK canonical text (not delta-accumulated) to match cryptographic signatures. Store signatures for replay.
- **Compaction robustness**: Force summarization when pruning is ineffective (<10% reduction). Prevent context overflow.
- **Discuss transcript clarity**: Interleaved format preserves conversation flow. Skip for participants (avoid redundancy with session history).
- **Cron session continuity**: Persist session IDs across daemon restarts (jobs reuse same session, context accumulates).
- **Error visibility**: Lock timestamps to first occurrence, enrich with slog attributes (session_id, step, error). Better debugging.
- **Max tokens recovery**: Auto-resume instead of fatal error. LLM can recover (use tools, outline-first strategy).

**Key insights**:
- **Thinking blocks are cryptographically signed**: Any JSON round-trip (Python `json.dumps`, character encoding normalization) breaks signatures. Must preserve exact bytes from SDK.
- **API auto-ignores old thinking blocks**: No need to strip at conversion time (API handles it). Pruner strips for disk cleanup only.
- **Compaction estimates are stale**: Post-prune token estimate uses last API call's token counts. Next call may overflow if estimate was too optimistic. Force summarization on ineffective pruning.
- **Discuss transcripts are shared context**: Captain already has discussion in session history (reused session). Transcript in `<previous_results>` is redundant for participants, essential for non-participants.
- **Subpane cleanup needs anim ticks**: Main chat `IsActive()` stops anim ticks, preventing idle subpane cleanup. Keep ticks running as long as subpanes exist.
- **Error timestamps drift**: `SetLastError` called on every snapshot update, re-stamping time. Lock to first occurrence by comparing stripped message text.

**Assumptions**:
- Opus 4.6 is primary model (extended thinking supported, 200K context)
- Session agent benefits from thinking (complex reasoning), subagents run faster without it
- Users tolerate cache misses from thinking mode switches (infrequent)
- Thinking content is private (no need to display in TUI)
- Discuss participants reuse sessions across stages (captain's session carries conversation history)
- Cron jobs are long-running background tasks (session continuity essential)
- Error bar shows latest error only (not a log, just current state)

**Blockers/risks**:
- **Thinking block corruption from manual edits**: Any Python/jq JSONL rewrite breaks signatures. Must use Go pruner or strip reasoning before rewrite.
- **Compaction overflow on edge cases**: If summary generation also hits max tokens, session is unrecoverable without manual truncation.
- **Discuss transcript size**: Interleaved format may be verbose. If discuss runs for many turns, transcript could inflate context for downstream stages.
- **Submodule sync overhead**: Two repos to push/pull. Memories updates must be committed/pushed separately from parent repo.
- **Hidden session recovery**: If user forgets session ID, no UI to list hidden sessions (must use `/session <id>` directly). Could add `/sessions all` command.

## Strategy & Approach

**Overall approach**: Implement cross-session discuss for context continuity, delegate tool for tactical flexibility, fix all audit-identified bugs, improve UX (token display, auto-close, sync animation).

**Why this approach**:
- **Cross-session > shared session**: Each agent keeps full context across stages. Discuss messages persist in agent sessions for downstream reasoning. No separate discuss session to manage/checkpoint.
- **Single-turn delegate > multi-turn**: Simpler implementation (no direct provider calls, no message surgery). Captain can chain calls for multi-turn effect. V2 can add multi-turn if needed.
- **ExtraTools > global registry**: Delegate needs engine reference (can't be static). Runtime injection clean, no interference with other sessions.
- **Pending inject > separate message**: Solves role-alternation issue (can't have consecutive user messages). Merged into next stage prompt as single user message.
- **For-tag warning > auto-delegate**: Teaches LLM correct tool usage. Auto-delegate would silently execute (confusing). Warning makes intent explicit.
- **EventSpawnDone per stage > per workflow**: Subpanes close between stages (visual feedback), re-open if agent used again. Prevents stale subpanes accumulating.
- **Centralized tick > per-pane toggle**: Eliminates desync from event-driven state changes. All panes blink together, cleaner UX.

**Key insights**:
- **Discuss cross-injection preserves role alternation**: `[speaker]\ntext` prepended to turn prompt = single user message. No separate injection needed.
- **Participant filter prevents redundancy**: Captain has full discuss in session history. `<previous_results>` would duplicate it. Filter skips for participants, includes for non-participants.
- **Pending inject solves stop-message timing**: Can't inject sage's `<done>` as separate message (breaks alternation). Can't skip it (captain loses context). Prepending to next stage prompt solves both.
- **Gate round overwrite race**: `markStage` sets `in_progress` with `GateRound: 0` before we read checkpoint's saved round. Must read first, then mark.
- **Discuss reconstructResult needs special handling**: Agent sessions have individual messages, but result needs `"discussion"` key with interleaved transcript. Reconstruct from session messages + `Participants` list.
- **Delegate enum needs sorting**: Workflow agents map iteration is random. Enum order affects tool schema → cache key. Sort for determinism.
- **For-tags in discuss create unclear intent**: Captain says `<for agent="sage">` but sage is the next speaker anyway. Warning clarifies: use `delegate` for out-of-turn, let discuss flow naturally for in-turn.

**Assumptions**:
- Captain is always first speaker in discuss (line 1052: `first := stage.Agents[0]`)
- Stop signal only checked on non-first agents (captain doesn't need to stop itself)
- Delegate tool only useful for captain (other agents don't orchestrate)
- Session agent has todolist tool (captain inherits it via `base: captain`)
- Logos workflows have ≤10 agents (delegate enum not huge)
- Subpanes idle timeout (1min) acceptable UX (not too aggressive)
