## Technical Context

**Architecture decisions**:
- **Extended thinking modes**: Anthropic supports `adaptive` (Claude decides), `enabled` (fixed budget), and disabled. For Opus 4.6: adaptive is recommended (preserves cache, allows interleaved thinking). Switching modes mid-session invalidates message cache breakpoints.
- **Thinking budget**: Only needed for `enabled` mode (min 1024, < max_tokens). `adaptive` has no budget param.
- **Cache invalidation from thinking**: Changing thinking parameters breaks message-level cache. Once set for a session, should stay consistent.
- **Agent defaults**: Session agent (long, complex reasoning) → `adaptive`. Lightweight agents (explore, condense, title) → `off` (no thinking overhead).
- **Slash command design**: `/think [adaptive|off]` to toggle (not mid-run, only between runs). Could also support `/think on` as alias for `adaptive`.

## Strategy & Approach

**Overall approach**: Enable extended thinking with cache-aware defaults. Make it configurable per-agent via config, but default to `adaptive` for complex reasoning agents (session) and `off` for lightweight agents (explore, condense). Expose a slash command for runtime toggling between runs.

**Why this approach**:
- `adaptive` mode is recommended by Anthropic for Opus 4.6 (preserves cache, supports interleaved thinking between tool calls)
- Fixed budget mode (`enabled`) isn't useful for agentic workflows on Opus 4.6 (no interleaved thinking)
- Switching modes mid-session breaks cache → restrict command to between-runs only, or accept the cache penalty
- Per-agent defaults allow fine-tuning (session agent benefits from thinking, title agent doesn't need it)

**Key insights**:
- **Cache invalidation is the main constraint**: Changing thinking parameters breaks message-level cache. Once a session starts with `adaptive`, it should stay that way. The `/think` command is for starting fresh runs with different modes, not toggling mid-run.
- **Opus 4.6 specific**: `adaptive` is the only practical mode for agentic use. `enabled` mode loses interleaved thinking. So the real choice is `adaptive` vs `off`.
- **Cost vs quality trade-off**: Extended thinking adds tokens (thinking content is billed). But for complex reasoning tasks (phase 3 opposition theory analysis), the quality improvement likely outweighs cost.
- **Thinking content is private**: Extended thinking blocks are not shown to the user (only in dev mode). The UI already has `OnReasoningDelta` wired but unused — could display thinking deltas in a debug pane.

**Assumptions**:
- Opus 4.6 is the primary model (extended thinking supported, adaptive mode recommended)
- Session agent is the main consumer of extended thinking (subagents run faster without it)
- Users prefer cache stability over mid-session mode switching (accept that changing `/think` invalidates cache)
- Thinking content doesn't need to be displayed (private reasoning, final output is what matters)

**Blockers/risks**:
- **Cache invalidation from mode switch**: If user runs `/think off` mid-session, all message cache is lost. Need to document this clearly or block mid-run changes.
- **Cost impact untested**: No empirical data on how much thinking adds to token usage. Could be significant for long sessions.
- **Haiku 4.5 doesn't support thinking**: If a session switches to Haiku (via `/model` command if it exists), thinking params must be stripped. Need to check model capabilities before setting params.
- **Thinking budget edge cases**: If we add `enabled` mode support later, need to validate `budget_tokens < max_tokens` and `>= 1024`.
