## Technical Context

**Architecture decisions**:
- **Cross-session discuss**: Each agent maintains persistent session across all stages. Discuss doesn't create shared session — agents see each other's responses via cross-injection (prepended to turn prompt). Clean session continuity, no redundancy.
- **Delegate tool via ExtraTools**: Runtime injection (not global registry). Only captain gets it. Needs engine reference (can't be static). Single-turn for v1 (captain can call multiple times for multi-turn effect).
- **Pending inject pattern**: Discuss stop signal stores final message in `pendingInject[agent]` map. Next stage's `stagePrompt` prepends it. Solves role-alternation issue (can't inject as separate message) and missing-context issue (captain needs sage's `<done>`).
- **Stage mode labels**: `[Stage: name (mode)]` prepended to all prompts. Captain knows whether to use gate tags. Discuss framing includes stage name for context.
- **For-tag auto-delegate**: Warn + re-prompt instead of auto-executing. LLM learns to use correct tool. Warning calls RunTagged which adds messages to session — `lastResponse` re-read afterward to capture corrected output.
- **EventSpawnDone per stage**: Logos stages emit spawn-done after each stage completes. Subpanes transition to idle, auto-close after 1min. If agent used in next stage, subpane re-created.
- **Centralized anim tick**: All panes derive `dotVisible` from `app.animTick % 2`. No per-pane state. Prevents desync from event-driven resets.
- **Participant filter**: Discuss results skipped in `<previous_results>` for participants (they have full conversation in session). Non-participants get transcript. Avoids redundancy.
- **JSONL compact**: Single-line messages preserve thinking signatures (no JSON round-trip character escaping). Easier manual inspection with `jq -c`.

**Patterns**:
- **ExtraTools injection**: `ls.ExtraTools = []tools.Tool{...}` before RunTagged. Runner merges with agent tools.
- **Pending inject**: `e.pendingInject[agent] = text` at source, `stagePrompt` checks and prepends, clears after use.
- **Cross-session message**: Prepend `[speaker]\ntext` to next agent's turn prompt. Single user message, no separate injection.
- **Stop signal**: `stopped = true` breaks inner loop, outer loop exits if `stopped`. Final `lastResponse` captured before break.

## Strategy & Approach

**Overall approach**: Implement cross-session discuss for context continuity, delegate tool for tactical flexibility, fix all audit-identified bugs, improve UX (token display, auto-close, sync animation).

**Why this approach**:
- **Cross-session > shared session**: Each agent keeps full context across stages. Discuss messages persist in agent sessions for downstream reasoning. No separate discuss session to manage/checkpoint.
- **Single-turn delegate > multi-turn**: Simpler implementation (no direct provider calls, no message surgery). Captain can chain calls for multi-turn effect. V2 can add multi-turn if needed.
- **ExtraTools > global registry**: Delegate needs engine reference (can't be static). Runtime injection clean, no interference with other sessions.
- **Pending inject > separate message**: Solves role-alternation issue (can't have consecutive user messages). Merged into next stage prompt as single user message.
- **For-tag warning > auto-delegate**: Teaches LLM correct tool usage. Auto-delegate would silently execute (confusing). Warning makes intent explicit.
- **EventSpawnDone per stage > per workflow**: Subpanes close between stages (visual feedback), re-open if agent used again. Prevents stale subpanes accumulating.
- **Centralized tick > per-pane toggle**: Eliminates desync from event-driven state changes. All panes blink together, cleaner UX.

**Key insights**:
- **Discuss cross-injection preserves role alternation**: `[speaker]\ntext` prepended to turn prompt = single user message. No separate injection needed.
- **Participant filter prevents redundancy**: Captain has full discuss in session history. `<previous_results>` would duplicate it. Filter skips for participants, includes for non-participants.
- **Pending inject solves stop-message timing**: Can't inject sage's `<done>` as separate message (breaks alternation). Can't skip it (captain loses context). Prepending to next stage prompt solves both.
- **Gate round overwrite race**: `markStage` sets `in_progress` with `GateRound: 0` before we read checkpoint's saved round. Must read first, then mark.
- **Discuss reconstructResult needs special handling**: Agent sessions have individual messages, but result needs `"discussion"` key with interleaved transcript. Reconstruct from session messages + `Participants` list.
- **Delegate enum needs sorting**: Workflow agents map iteration is random. Enum order affects tool schema → cache key. Sort for determinism.
- **For-tags in discuss create unclear intent**: Captain says `<for agent="sage">` but sage is the next speaker anyway. Warning clarifies: use `delegate` for out-of-turn, let discuss flow naturally for in-turn.

**Assumptions**:
- Captain is always first speaker in discuss (line 1052: `first := stage.Agents[0]`)
- Stop signal only checked on non-first agents (captain doesn't need to stop itself)
- Delegate tool only useful for captain (other agents don't orchestrate)
- Session agent has todolist tool (captain inherits it via `base: captain`)
- Logos workflows have ≤10 agents (delegate enum not huge)
- Subpanes idle timeout (1min) acceptable UX (not too aggressive)
- JSONL compact doesn't break external parsers (single-line is standard JSONL)

**Blockers/risks**:
- **Delegate abuse**: Captain could spam delegate calls, blowing up context/cost. No rate limiting. Mitigated: LLM usually self-regulates, workflows are short.
- **Pending inject stale data**: If captain doesn't run in next stage, pending inject sits forever. Low risk: captain usually in review stage after discuss.
- **Cross-session context explosion**: Each agent accumulates full workflow history. Mitigated: compaction runs, cache reuse helps.
- **For-tag false positives**: Captain might legitimately use `<for>` in discuss (directing next speaker). Warning fires incorrectly. Low impact: just a re-prompt.
- **Stop signal parse failures**: If sage's output is truncated or malformed, `<done>` might not parse. Discuss runs until max turns. Acceptable: turn limit prevents infinite loop.
- **Discuss transcript size**: Interleaved format verbose. If discuss runs 10+ turns, transcript inflates `<previous_results>` for non-participants. Mitigated: participant filter reduces recipients.

## Strategy & Approach

**Overall approach**: Fix discuss output structure (speak tool), fix UI bugs (animation, context cancel, sizing), refactor checkpoint to cursor-based (robust resume), add proactive token refresh, add server error retry, fix thinking block ordering.

**Why this approach**:

1. **Speak tool > text parsing**: Structural output prevents framing pollution, explicit stop signal replaces fragile `<done>` tag. LLM learns correct tool usage via prompting + terminal behavior.

2. **Centralized anim tick > per-pane toggle**: Eliminates race conditions from event-driven state changes. All panes blink together, simpler mental model.

3. **Context swap > shared context**: Old runLoop must survive disconnect. Swapping context before detach cleanest solution. Matches existing `Detach()` pattern.

4. **Cursor checkpoint > per-stage status**: Single source of truth (cursor), not inferred (isStageComplete heuristic). Results stored directly, not reconstructed from session last-messages. Gate loop blind spot eliminated. Resume trivial.

5. **Proactive refresh > reactive only**: Prevents workflow disruption from auth expiry during long-running bash tools. Self-scheduling based on actual expiry time simpler than cron. `UpdateAuth` swap cleaner than process restart.

6. **Logos retry > provider retry only**: Higher-level retry (30s delays) gives transient outages time to clear. Complements provider-level retry (10s delays). Server errors only to avoid burning budget on permanent failures.

7. **Preserve block order > reordering**: Signature validation tied to original position. Reordering breaks API validation. Streaming order matches API's original structure.

**Key insights**:

- **Speak tool Terminal: true ends turn cleanly** — no need for separate stop detection loop. Fallback to `extractLastText` preserves compatibility.

- **Animation desync from duplicate chains** — each `startAnimTick()` launches independent chain. Session switch triggered second launch without stopping first. `animRunning` bool prevents duplicates.

- **Context cancel propagates to detached sessions** — old runLoop's captured `connCtx` gets cancelled on disconnect. Swapping context before detach preserves old goroutine.

- **Gate loop was checkpoint blind spot** — inner loop re-ran stages but never updated checkpoint. Resume logic operated on stale data from first pass. `setCursor` at every boundary eliminates gap.

- **Cursor replaces inference** — no more "walk stages and figure out where we were". Checkpoint explicitly says where to resume. `isStageComplete` heuristic deleted.

- **Proactive refresh prevents workflow disruption** — long-running bash tools (python scripts) can exceed token TTL. Reactive refresh only triggers after failure. Proactive refresh prevents disruption.

- **Provider retry insufficient for transient outages** — 10s delays too short for server recovery. 30s logos-level retry gives outages time to clear.

- **Thinking block reordering breaks signatures** — API validates signatures against original position. Grouping thinking blocks first changes positions. Preserving streaming order maintains validation.

**Assumptions**:

- Speak tool only useful for discuss phases (solo agents don't need it)
- Animation tick rate (500ms) acceptable for all panes
- 1min idle timeout before subpane close acceptable UX
- Gate iterations ≤ 32 typical (checkpoint cursor can track)
- OAuth token refresh 10min before expiry sufficient buffer
- Server errors (500+) transient, client errors (400) permanent
- Thinking block signatures tied to position (validated empirically)

**Blockers/risks**:

- **Speak tool learning curve** — LLMs must learn to call speak. Fallback mitigates but may produce verbose transcripts if LLM outputs text + calls speak.

**Assistant** (2026-03-15 17:46):
Here's how it works:

**Per-agent YAML flags**: `contribute: true` (reads flow out), `receive: true` (reads flow in).

**Flow**:
1. After each stage completes, `collectStageReads` scans messages from `contribute` agents — finds `read` tool calls, matches them to tool results, puts file path→content into a `readCache` (LRU, 40k char budget).

2. Before each stage runs, `runStage` calls `injectSharedReads` on each `receive` agent — builds a `<shared_file_reads>` context file block with all accumulated reads, injected into `ContextFiles` (same system prompt slot as project context). Idempotent — replaces existing entry on repeat.

3. The cache is cumulative across stages — reads from stage 1 are still available in stage 3. LRU eviction drops oldest when budget exceeded.

**Effect**: Captain reads files → coder/sage automatically get those file contents in their system prompt without re-reading. Avoids redundant tool calls across agents.
