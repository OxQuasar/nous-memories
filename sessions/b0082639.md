## Flashbots

Flashbots is a private transaction relay system that sits between MEV searchers and block builders/validators on Ethereum. It was created to address the negative externalities of MEV extraction — gas wars, failed transactions clogging the chain, and frontrunning.

### The Problem It Solved

Before Flashbots, MEV searchers competed by bidding up gas prices in the public mempool. This caused:
- Gas spikes for all users during MEV opportunities
- Failed txs that still consumed gas (searchers submitting competing txs, only one wins)
- Frontrunning via public mempool visibility

### How It Works

**Traditional flow:**
```
User tx → public mempool → validators pick highest gas → block
```

**Flashbots flow:**
```
Searcher → bundles txs → Flashbots Relay → Builder → Proposer (validator)
```

Key components:

1. **Searchers** — bots that find MEV opportunities (arb, liquidations, etc.) and construct "bundles" — ordered sets of transactions that must execute atomically and in sequence.

2. **MEV-Share / MEV-Boost Relay** — a private channel. Bundles are sent here instead of the public mempool. Failed bundles don't hit chain, don't cost gas. The relay runs an auction.

3. **Builders** — aggregate bundles from multiple searchers, combine with regular user txs, and construct full blocks optimized for total extracted value.

4. **Proposers (validators)** — receive completed blocks from builders via the relay. They pick the highest-value block. The builder pays the proposer a "bid" (tip) for inclusion.

### Key Properties

- **No public mempool exposure** — searcher strategies stay private. Nobody can frontrun the frontrunner.
- **No failed tx cost** — if a bundle doesn't land, nothing is spent. Simulation happens off-chain first.
- **Atomic bundles** — txs execute together or not at all. Searchers can compose multi-step arb into one bundle.
- **Sealed-bid auction** — searchers bid for inclusion by sharing profit with builders/proposers. Competition drives most of the profit upstream — data suggests searchers often pay 90%+ of revenue to proposers.

### PBS (Proposer-Builder Separation)

Flashbots drove the adoption of PBS on Ethereum post-merge:
- **Proposers** (validators) no longer build blocks themselves
- **Builders** specialize in block construction and MEV optimization
- This separates the "what goes in the block" decision from the "who validates" decision

As of 2026, Ethereum is working on **ePBS** (Enshrined PBS) via the Glamsterdam upgrade — baking this separation directly into the protocol rather than relying on Flashbots as external infrastructure.

### MEV-Share

A newer Flashbots innovation where MEV profit is partially **kicked back to the user** whose transaction created the opportunity. The user's order flow is shared with searchers, but the user gets a rebate. Turns the "invisible tax" into a partial refund.

### Why This Matters for Freya

Freya operates at block-level only — no mempool, no Flashbots integration. It:
- Sees opportunities **after** they're already visible to everyone
- Submits txs to the public mempool via raw RPC broadcast
- Has no bundle guarantees — its tx can be frontrun or fail

On any chain with active Flashbots/PBS infrastructure (Ethereum, and increasingly L2s), a Freya-style bot is structurally disadvantaged. The profitable arb opportunities are captured in the builder auction before they ever reach the next block's public state. Freya would only find leftovers — opportunities too small for the MEV supply chain to bother with, or ones that emerge between blocks on chains without this infrastructure.
